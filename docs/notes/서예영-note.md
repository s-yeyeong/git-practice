# Git & GitHub 협업 워크플로우 개념 정리 노트

## 1. 기본 용어

워크플로우 (Workflow): 작업이 진행되는 전체 과정 또는 절차

이슈 (Issue): 해야 할 일 또는 작업 요청서 (우리가 해결해야 할 Task 단위)

유틸 함수 (Util Function): 특정 클래스에 속하지 않고 여러 곳에서 재사용할 수 있는 보조 함수

리팩토링 (Refactor): 프로그램의 기능은 바꾸지 않고, 코드 구조와 가독성을 개선하는 작업

브랜치 네이밍 규칙: 브랜치 이름을 일정한 형식으로 짓는 규칙 (예: feature/이름-작업명)

## 2. GitHub Flow 

GitHub에서 사용하는 가장 기본적이고 강력한 브랜치 전략

main 브랜치 (항상 배포 가능한 완벽한 상태 유지)

새 브랜치 생성 (feature/..)

기능 개발 (내 브랜치에서 안전하게 작업)

Commit & Push (작업 내용 저장 후 원격에 올리기)

Pull Request (PR) (내 코드를 main에 합쳐달라고 요청 및 리뷰 시작)

코드 리뷰 (팀원들과 코드 피드백)

main 브랜치에 Merge (승인 완료 후 최종 병합)

배포

## 3. 브랜치 병합과 충돌

Merge (머지): 두 브랜치의 변경 내용을 하나로 통합하는 작업

Rebase (리베이스): 한 브랜치의 작업(커밋들)을 다른 브랜치의 최신 상태 위로 잘라 붙여서 다시 올리는 것 (히스토리가 깔끔해짐)

hunk (인접 라인): Git이 변경 사항을 비교할 때 보여주는 '하나의 변경 덩어리' (인접한 변경 라인들의 묶음)

비자명 충돌 (Non-trivial Conflict): 컴퓨터(Git)가 자동으로 해결하지 못해, 사람이 직접 개입해서 어떤 코드를 남길지 결정해야 하는 충돌

## 4. Git 트러블슈팅 필수 명령어 5대장

- git commit --amend -m "내용"

최근 커밋 메시지 수정

방금 한 커밋의 메시지에 오타가 났거나 내용을 덧붙일 때

- git reset --soft HEAD~1

최근 커밋 취소 + 작업 내용은 유지

커밋은 취소하고 싶지만, 내가 수정한 코드 파일은 그대로 남기고 싶을 때

- git revert <커밋해시>

원격에 Push된 커밋 취소

이미 GitHub에 올라가 버린 커밋을 안전하게 취소하고 싶을 때 (취소했다는 새로운 기록이 남음)

- git stash

현재 작업을 임시 보관

작업 도중 급하게 다른 브랜치로 넘어가야 하는데, 아직 커밋하긴 애매할 때

- git stash pop

임시 보관한 작업 다시 꺼내기

다른 볼일을 마치고 원래 브랜치로 돌아와서 하던 작업을 마저 이어갈 때
